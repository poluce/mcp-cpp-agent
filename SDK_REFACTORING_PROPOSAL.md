# MCP C++ Agent SDK 重构改进建议

基于 `multi_server_agent` 示例的实测与排查，当前 `src` 目录下的 MCP SDK 暴露出了一些工程化设计上的问题。这些问题主要集中在“导致主界面卡死”、“代理配置繁琐泄露”以及“异常处理陷入死循环”这三个方面。

为了在保证功能的前提下，遵循 **KISS (Keep It Simple, Stupid)** 原则，拒绝过度设计，我们提出以下三点精简而致命的重构建议。这些建议旨在通过最小的代码变动，彻底根除 SDK 的不适感。

---

## 1. 改进点一：将系统代理能力内置到底层 (解决配置别扭)

### 诊断问题
当前版本中，当启动依赖海外网络的子进程（如基于 Node.js/npx 的 `anysearch` MCP 服务）时，由于子进程无法连通网络，导致握手失败。
虽然在应用层（`McpServerManager.cpp`）手动抓取 `QNetworkProxyFactory` 并注入给子进程环境变量能解决问题，但这属于典型的**抽象泄露（Leaky Abstraction）**。要求每一个使用本 SDK 的业务层都去手写一遍代理注入逻辑，非常别扭且不合理。

### 建议方案
将代理注入的代码下沉，直接封装在 SDK 内部负责拉起子进程的类中：`src/transport/src/QtProcessStdioTransport.cpp`。
在该类的 `start()` 或环境初始化方法中，自动探测系统当前的 HTTP/SOCKS5 代理，并隐式注入到 `QProcess` 的环境变量（`HTTP_PROXY`, `HTTPS_PROXY`）中。

### 预期效果
- 业务层**零配置**即可获得穿透代理的能力。
- 符合“开箱即用”的现代 SDK 标准。

---

## 2. 改进点二：拆除“死循环定时炸弹” (解决卡死崩溃)

### 诊断问题
在 `McpQtClient::handleTransportFailure` 方法中，目前的重连机制极其粗暴。它使用了一个 `while` 循环嵌套同步等待的方式去不断尝试重连。一旦网络环境彻底断开，或者目标服务端未启动，整个 Qt 程序的主线程就会永远困在这个死循环中，导致程序界面完全卡死（无响应）。

### 建议方案
废除阻塞式的死循环，改用纯异步的事件驱动：
利用 Qt 最基础的 `QTimer::singleShot`（单次定时器）。当检测到断开时，抛出一个“3秒后执行重连”的定时任务，然后函数直接 `return` 结束，将控制权交还给主事件循环。

### 预期效果
- 无论服务端如何崩溃、网络如何离线，**主界面永远不会卡死**。
- 代码逻辑更加扁平清晰，消除不可控的死锁风险。

---

## 3. 改进点三：拥抱纯异步，移除主线程阻塞 (解决等待卡死)

### 诊断问题
SDK 在向远端 MCP 服务请求数据时（例如 `fetchAllTools()` 获取工具列表），错误地利用 `QEventLoop::exec()` 将本该异步的网络请求包装成了同步等待。
这就导致：如果远端服务响应缓慢，主线程就会被强行挂起（这也是此前不得不为 `anysearch` 硬编码跳过逻辑的根本原因）。

### 建议方案
采用经典的 **Signal-Slot (信号与槽)** 异步设计：
1. 废弃在 `fetchAllTools` 等方法中的 `QEventLoop` 阻塞写法。
2. 客户端向远端发送 `tools/list` 请求后立刻返回。
3. 当底层的 JSON-RPC 真正收到远端回包并解析完毕后，通过 `McpQtClient` 抛出一个标准的 Qt 信号，例如 `void toolsReady(const QList<McpTool>& tools)`。
4. 外层（如 `AgentSession`）连接该信号，当收到 `toolsReady` 后再触发下一步流程。

### 预期效果
- 彻底消灭 SDK 内部的假性死锁。
- 真正实现事件驱动的高性能 IO，让代理层即使同时应对数十个不同响应速度的 MCP 服务也能游刃有余。
